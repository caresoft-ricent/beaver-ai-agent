## 1. 文档结论先行

当前已经形成的统一判断是：

### 1.1 河狸云 Java 主系统定位不变
河狸云继续作为：
- 正式业务系统
- 正式数据来源
- 正式权限系统
- 正式流程系统
- 正式 API / Gateway 提供方
- 正式报表与台账输出方

### 1.2 beaver-ai-agent 不应被理解为河狸云外挂聊天页
beaver-ai-agent 的正确定位是：
- AI Kernel
- 语义层
- 能力层
- 规则层
- 执行层
- 智能入口 / 智能工作台

### 1.3 当前核心问题不在“AI 够不够聪明”
而在于：
- 河狸云 Java 侧还没有形成正式的 AI 接入底座
- 上下文、票据、网关、元数据、scope、权限校验都还不够正式
- 导致 Kernel 现在很多配置是在“补系统缺口”

### 1.4 下一阶段最重要的方向
不是让 Java 团队继续分散做 AI 功能，而是：

# **把河狸云改造成一个可被 beaver-ai-agent 稳定接入、可信调用、权限可控、结果标准的业务能力系统。**

## 2. 当前河狸云 Java 侧现状

---

## 2.1 当前主系统技术现状
河狸云当前核心业务系统采用：
- Java / Spring 技术栈
- 多租户业务模式
- 多模块业务系统
- 正式权限、组织、成员关系和业务对象都在 Java 主系统侧

这个判断一直没变：

### 适合作为业务主系统
没问题。

### 不适合作为 AI 主底座继续分散扩写
这是当前已经形成的共识。

原因不是 Java 不能做 AI，而是：
- AI Runtime、Tool/Capability 编排、语义层、规则层、快速试错节奏，更适合单独沉淀在 beaver-ai-agent
- 如果继续在 Java 里零散堆 AI 主逻辑，会越来越难扩展、越来越难维护

---

## 2.2 当前 Java 侧在 AI 接入上存在的现实问题

### 问题 1：认证 / OAuth2 / SSO 还不稳定或未形成可验证闭环
你已经明确说过：
- 包括 OAuth2 和 SSO 在内，Java 侧当前进展“还很糟糕”
- 不是简单“配上了”，而是没有形成可验证、可联调、可用的真实闭环

这意味着：
- Kernel 不能稳定依赖 Java 侧的正式单点入口
- 当前阶段不能把复杂 SSO 当成先决条件
- 只能先采用更轻量的 identity exchange / ticket 方式来过渡

---

### 问题 2：上下文仍然不是正式 ticket，而偏前端态拼接
从前面对现状的分析，已经形成过明确判断：

当前很多上下文仍然来自：
- 前端 localStorage
- header
- referer
- 浏览器态中的企业/用户信息

这导致：
- 上下文不够正式
- 可控性不强
- 不适合长期平台化
- Kernel 需要用大量配置弥补

---

### 问题 3：Java 侧还没有正式 Capability Gateway
当前很多能力接入，本质仍然是：
- AI 侧猜接口
- Connector 侧拼 header
- 场景侧找 action
- 技术上更多是散接口接入，而不是标准 capability 接入

这会带来：
- 每加一个场景都要重新理接口
- 参数、字段、错误码和权限都容易不一致
- Kernel 很难真正平台化

---

### 问题 4：多租户/多 membership 关系虽然业务上存在，但 AI 侧契约还没正式化
你已经补充过一个非常关键的现实情况：

- 一个河狸云账号，可能在多个 enterprise / tenant 下有权限
- 当前租户/企业/模块是前端切换产生的 active scope
- 这不是“一个用户 = 一个固定租户”的系统

这意味着：
- Java 侧必须把 membership / active scope 作为正式概念给 AI
- 不能只给一个当前 tenant 或 enterprise
- 不能只靠前端切换页面状态而不形成正式协议

---

### 问题 5：元数据、对象关系、字段语义还没有以 AI 友好的方式正式输出
虽然河狸云本身已有业务对象，但当前问题是：

- 对象怎么被 AI 理解
- 字段如何统一语义
- 状态如何标准解释
- 对象关系如何正式暴露
- 能力如何分组和命名

这些还没有完全形成 AI 友好的正式输出层。

所以现在 Kernel 侧很多工作量并不是“重复造轮子”，而是在：
- 做语义映射
- 做 canonical schema
- 做 capability 定义
- 做规则输入输出口径

---

## 3. 对河狸云 Java 侧的正确定位

---

## 3.1 不是 AI 主引擎
已经形成明确共识：

### Java 团队后续不应继续分散实现：
- 长提示词主逻辑
- 意图识别主链
- Agent 编排主链
- 本体主模型
- AI workflow 主引擎

这些都应统一回到 beaver-ai-agent。

---

## 3.2 是“业务能力宿主系统”
河狸云 Java 侧后续应该重点承担的职责是：

### A. 身份与上下文来源
给 AI 提供：
- 用户身份
- memberships 列表
- 默认 scope
- 当前 scope 切换后的可信信息

### B. 正式能力提供方
给 AI 提供：
- 标准化 capability gateway
- 正式业务事实
- 正式业务动作能力

### C. 元数据与对象来源
给 AI 提供：
- 企业/组织/项目/区域字典
- 模块字典
- 状态枚举
- 业务对象基本定义

### D. 权限与数据域校验方
给 AI 提供：
- 谁能看
- 谁能做
- 哪些范围可查
- 哪些写操作必须确认

一句话：

# **Java 侧不是 AI 主引擎，而是 AI 可稳定接入的业务宿主系统。**

## 4. beaver-ai-agent 当前定位

---

## 4.1 当前不是要替代河狸云
而是做：
- 独立的 AI Kernel
- 独立的 AI 运行时
- 独立的 Session / Scope Runtime
- 独立的 Scenario / Capability / Evidence / Policy 层

### 主入口建议
长期建议：
- 独立域名：`ai.ricent.com`

短中期仍可以：
- `beaver-ai-agent.ricent.com`

### 使用方式
- 用户可直接打开 AI 对话入口
- 也可嵌入河狸云页面
- 无论哪种入口，后端执行模型都一致

---

## 4.2 当前已形成的共识
beaver-ai-agent 后续要统一维护：

- `identity`
- `memberships[]`
- `active_scope`
- `Session Runtime`
- `Capability Runtime`
- `Evidence`
- `Policy`
- `Scenario`
- `Domain Pack`
- `Adapter / Mapping`

也就是说：

### Java 侧负责“事实与能力源”
### AI 侧负责“理解、执行、解释与智能入口”

## 5. 河狸云 Java 与 beaver-ai-agent 的未来对接主线

这一部分是整个规划最关键的主线。

---

## 5.1 主线 1：Identity Exchange / Context Ticket

### 目标
不再长期依赖：
- localStorage
- referer
- 裸 header
- 前端当前页面细节

而是让 Java 侧正式提供：

- identity
- memberships
- default_scope

### 正确理解
ticket 不再是“当前企业票据”，而是：

# **身份 + 可选 memberships + 默认 scope 的正式票据**

### 至少应包含
- user_id
- username
- display_name
- memberships[]
- default_scope
- issued_at
- expires_at
- signature

### 作用
- 让 Kernel 建立自己的独立 session
- 让 Kernel 不被河狸云前端实现细节绑定
- 为以后接第二个系统做准备

---

## 5.2 主线 2：Membership / Active Scope 正式化

### 关键判断
这不是简单“切企业”，而是：

# **一个账号拥有多个 memberships**
# **当前运行时上下文由 active_scope 表达**

### 为什么不是只切 enterprise
因为真实变化的不只是：
- enterprise_id

还包括：
- member_id
- org_id
- ouid
- role_type
- region/project
- current_module

### Java 侧未来要提供
至少要让 AI 可以知道：
- 当前有哪些 memberships
- 默认 scope 是什么
- 当前 active scope 合法切换依据是什么

---

## 5.3 主线 3：Capability Gateway

### 当前现实问题
现在 AI 侧很多时候还是：
- 猜接口
- 靠 connector 拼 header
- 按 action 零散调用

### 正确方向
Java 侧必须逐步收敛出：

# **标准 Capability Gateway**

例如：
- `POST /ai/capabilities/execute`

### 统一输入
- capability_code
- scope
- params
- request_id

### 统一输出
- success
- code
- message
- data
- evidence_payload
- confirm_required

### 作用
- 降低 AI 侧配置工作量
- 让能力真正以 capability 形式接入
- 统一错误码、权限和结果结构

---

## 5.4 主线 4：元数据接口正式化

Java 侧要逐步正式输出：

- 企业/项目/区域树
- 组织树
- 角色字典
- 模块字典
- 状态枚举
- 业务对象基础定义

这样 Kernel 才能从：
- 技术补洞型配置

转向：
- 业务表达型配置

---

## 5.5 主线 5：嵌入模式下的 scope 切换联动

### 当前结论
这部分可以等 Java 票据和网关准备好后，在联调时再实现。

### 但架构方向已经明确
嵌入模式下：
- 河狸云前端切换成功后，主动通知 Kernel
- 最终都收敛到：
  - `POST /session/activate-scope`
  - 或 `POST /session/exchange-scope-ticket`

### 必须坚持的规则
- 宿主前端只负责通知切换意图
- Kernel 后端负责重建并确认 active_scope
- 后续所有执行以后端 session 为准

而要优先做：

2. Identity / Context Ticket

### 首批建议能力
- issue.query

- contract.payment_analysis

## 8. 当前阶段 Java 侧最大价值，不在“多写一点 AI 代码”

当前 Java 侧最大的价值，在于：

### 从“河狸云业务系统”
升级成：

# **AI 可接入的正式业务宿主系统**

这意味着：
- 能被识别
- 能被安全调用
- 能提供稳定上下文
- 能提供标准能力
- 能被多系统、多场景复用

这才是对 beaver-ai-agent 长期最有价值的支持。

## 9. 和配置工作量的关系

已经形成明确判断：

### 现在配置工作量高
主要是因为：
- 上下文没正式打通
- 接口没正式收敛
- 字段和语义没正式输出

### 一旦 Java 侧按正确方式打通
配置工作量会明显下降，尤其下降的是：

- 上下文补洞
- header 补丁
- 接口差异适配
- 字典/状态人工解释

### 不会消失的
仍然会保留：
- 场景配置
- 规则配置
- 语义确认
- 输出模板

一句话：
- 低价值配置会下降
- 高价值配置会保留

## 10. 和远期愿景的关系

远期愿景已经明确是：

- 宿主对象与能力自动发现
- Ontology / Mapping / Capability / Scenario 草案自动生成
- 人工审核、修订、发布

### Java 侧在这个愿景中的作用
未来如果要做到：
- 机器辅助发现对象与能力
- 自动生成草案

那么 Java 侧越早做到下面这些，未来越容易：

#### 1. 接口标准化
#### 2. 元数据标准化
#### 3. 对象关系显式化
#### 4. 能力网关化
#### 5. 权限与数据域可表达化

所以：

### 当前 Java 侧打通，不只是为“当前能联调”
也是在为将来：
- 自动发现
- 半自动配置
- Builder / Review Studio

铺路。

## 11. 当前阶段最重要的执行顺序（可在新会话继续沿用）

### Java 侧优先顺序
1. 先把 identity / memberships / default_scope 正式给出来
2. 再给 Capability Gateway 首版
3. 再给元数据接口
4. 最后在联调阶段接嵌入模式的 scope 切换通知链路

### Kernel 侧优先顺序
1. 先补 Session Runtime
2. 先补 `/session/exchange` `/session/me` `/session/activate-scope`
3. 先做 `scope_proxy`
4. 再做独立前端的 Scope Switcher
5. 再接河狸云联调
6. 再用 Domain Pack 接高价值业务场景

---

## 12. 最后一句话总结

河狸云 Java 侧的未来对接规划，不是让它继续做 AI 主引擎，而是让它逐步成为：

# **beaver-ai-agent 的正式业务宿主系统、身份来源、能力来源和元数据来源。**

与此同时，beaver-ai-agent 继续作为：

# **独立 AI Kernel、独立 Session/Scope Runtime、独立智能工作台与长期平台化内核。**
