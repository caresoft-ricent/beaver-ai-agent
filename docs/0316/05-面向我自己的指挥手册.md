# 面向我自己的指挥手册

> 目标：明确我在这条路线中的角色、边界、指挥方式和决策准则，确保未来能持续指挥 AI 和团队，而不是被项目规模反过来拖住。

## 1. 先明确我的角色

- 河狸云 AI Kernel 总设计师
- 系统边界定义者
- 方法论制定者
- AI 协同开发指挥者
- 平台演进方向把关人
- 产品化与客户价值裁决者

我最重要的工作不是多写几段代码，而是：

1. 定义系统定位
2. 定义边界
3. 定义路线
4. 定义约束
5. 定义优先级
6. 决定哪些应该平台化，哪些只做项目交付

---

## 2. 我真正要指挥的，不是功能，而是"平台模式"

每个需求先判断属于哪一类：

### A. 平台能力建设
scope 模型、capability registry、policy layer、evidence layer、eval 体系。

### B. 场景模板建设
问题查询模板、问题关闭模板、概况查询模板、巡检分析模板。

### C. 单客户特化交付
某客户特有口径、特有字段、特有流程分支。

**决策原则：** 能沉淀为平台能力的优先平台化；能沉淀为模板的优先模板化；纯客户特例不要污染平台主干。

---

## 3. 我如何指挥 AI

### 3.1 不能给 AI 的指令方式

不要只说：帮我做这个、优化一下、把功能加上、效果不对改一下。

### 3.2 每次下任务必须包含 6 个要素

1. **目标**：这次要解决什么问题
2. **边界**：允许改哪些目录，不允许改哪些部分
3. **上下文**：当前项目结构、设计、路线位置
4. **输出物**：代码、migration、测试、文档、配置样例
5. **验收标准**：什么叫完成
6. **禁止事项**：明确哪些不能做

### 3.3 标准任务模板

```text
任务名称：
目标：
背景：
本次只允许修改：
本次禁止修改：
输出物：
1. 代码
2. 测试
3. 迁移说明
4. 文档
验收标准：
注意事项：
```

## 4. 我如何指挥团队

### 4.1 对 Java 团队

**第一阶段：不要求配合。** AI 侧自行从前端 Header 提取上下文，Java 团队零改动。

**第二阶段开始：** 要求提供 Capability Gateway 和统一错误码。

给 Java 团队的要求应是：
- 输出接口契约，不是临时接口
- 输出标准能力，不是页面专用接口
- 输出主规则和权限，不是让 AI 猜

### 4.2 对产品/项目经理

短期目标不是让他们"写提示词"或"自助配置"，而是：
- 理解业务对象、流程、能力边界
- 说清楚必填参数和可追问参数
- 说清楚哪些动作需要确认
- 说清楚客户的真实业务口径
- 验证 AI 输出是否正确

**真正的低代码配置平台是 Stage 4 的事，不是现在。**

### 4.3 对 AI 协作开发者/技术同事

要求的不是"先做出效果"，而是：先进入平台边界，再做功能实现，所有新增能力必须能沉淀。

## 5. 我必须定下来的边界规则

### 系统边界规则
1. Java 河狸云负责业务事实和正式执行；AI Kernel 负责理解、编排、解释
2. AI Kernel 独立部署、独立演进，不嵌入 Java 主系统内部
3. AI 与河狸云通过标准协议协作。第一阶段：Header 提取；第二阶段：Capability Gateway；第三阶段：Context Ticket

### 研发边界规则
4. 新增能力必须进入 Capability Registry，不允许直接堆在 engine 里
5. Skill 不是系统核心，是场景入口层（未来拆分为 Scenario + Playbook）
6. 任何关键上下文、输入、输出都必须 schema 化
7. 写操作能力必须有 policy 和 confirmation
8. 任何功能改动都必须有测试

### 产品边界规则
9. 优先做成熟、高频、边界清晰的场景
10. 不要为了展示效果，把平台主干污染成大量客户特例
11. 能模板化的一律模板化；能平台化的一律平台化

---

## 6. 我如何做优先级判断

### 第一问：这个需求能否沉淀为平台能力？
如果能，优先平台化。

### 第二问：这个需求是否适合作为模板场景？
如果能，优先模板化。

### 第三问：这个需求是不是只属于单客户特例？
如果是，做隔离，不污染主干。

### 第四问：这个需求是否要求改动系统边界？
如果要求，先评审边界，不急着开发。

### 第五问：当前在哪个 Stage？这个需求属于哪个 Stage？
如果不属于当前 Stage，放到后面，不提前做。

## 8. 我的四条纪律

### 纪律一：不被"快速效果"绑架
如果一个做法会破坏平台边界，即使短期看起来更快，也不能轻易接受。

### 纪律二：任何新增需求都先判断归属层
平台层 → 模板层 → 客户特例层。

### 纪律三：坚持让 AI 按边界工作
不是让 AI "尽快帮我做完"，而是让 AI "按平台规则帮我做对"。

### 纪律四：每个 Stage 必须同时交付至少一个可跑通的客户场景
不允许出现"架构升级做了三个月但客户侧无感知"的情况。架构升级和场景交付必须同步推进。

---

## 9. 我怎样避免自己陷入"亲自盯所有细节"的陷阱

### 我应该亲自抓的
- 架构、边界、路线
- 关键命名、核心模型
- 关键评审
- 客户价值判断
- Stage 完成度验收

### 我不应该长期亲自抓的
- 普通 CRUD
- 页面小改
- 单个接口对接细节
- 大量模板生成
- 普通测试补齐

这些应该逐步交给 AI、技术同事、产品/项目配置人员。

## 10. 我的自建 AI 基础设施如何与 AI Kernel 衔接

我的 Mac Studio M3 Ultra + vllm-mlx + Qwen3 系列是完整的推理基础设施。AI Kernel 应该利用这个优势：

**建立 LLMRouter 中间层：** 根据 task_type（intent / entity_extraction / response / summary）+ 延迟要求自动选择模型。意图识别和实体抽取用本地小模型（快、便宜），回复生成用更大的模型或云端 API（质量高）。

**Evidence 层是训练数据生产线：** 当前 EvidenceCollector 记录的完整决策链，清洗后就是微调垂直小模型的高质量 SFT 数据。建议在 Evidence 结构中增加 `quality_label` 字段（success / partial / fail），积累到一定量级后启动微调。

## 11. 给我自己的一句话结论

我未来真正要做的，不是亲自把所有 AI 功能做完，而是：

**把河狸云 AI Kernel 的地基打稳、边界立住、规则定清，然后一个 Stage 一个 Stage 地往上盖楼。每盖一层楼，同时交付一个客户可感知的场景。不急，但不停。**
