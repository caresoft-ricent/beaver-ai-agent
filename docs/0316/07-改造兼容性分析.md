## 2. 各 Stage 兼容性影响分析

### 2.1 Stage 0：地基固化（合并双引擎 + 补测试）

**数据库影响：无**

| 改造项 | 影响范围 | 兼容性风险 | 保障措施 |
|--------|----------|-----------|----------|
| 抽取 pipeline.py | 纯代码重构 | 中 — 逻辑等价性 | 先写测试建立基线，再重构 |
| engine.py 调用 pipeline | 同步引擎内部 | 低 | `/completions` 端点回归测试 |
| stream_engine.py 调用 pipeline | 流式引擎内部 | 低 | `/stream` 端点回归测试 |
| 建立 tests/ | 新增文件 | 无 | — |

**关键约束：**
- 不改任何表结构
- 不改 API 入参/出参格式
- 不改 SSE 事件协议
- engine.py 和 stream_engine.py 对外接口签名保持不变

**回滚方案：** git revert 即可，无数据迁移。

---

### 2.2 Stage 1：Scope 接入（BeaverSessionScope）

**数据库影响：可选，视方案而定**

| 改造项 | 影响范围 | 兼容性风险 | 保障措施 |
|--------|----------|-----------|----------|
| 新建 kernel/scope.py | 新增文件 | 无 | — |
| chat.py 提取 Scope 传入引擎 | API 入口层 | 中 | **渐进式：scope 为可选参数** |
| stream_dialog 增加 scope 参数 | 引擎签名 | 中 | 默认值 None，无 Scope 时走现有逻辑 |
| 日志记录 enterprise_id | evidence.py | 低 | 纯追加 |

**关键兼容性设计：渐进式 Scope 注入**

```python
# Stage 1 的 stream_dialog 签名
async def stream_dialog(
    db, tenant_id, customer_id, session_id, message,
    thread_id, run_id,
    scope: BeaverSessionScope | None = None,  # ← 新增，可选
):
```

- 有 Scope 时：用 scope.enterprise_id 替代 tenant_id 查配置，用 scope.token 代理调用 API
- 无 Scope 时：fallback 到现有 tenant_id + connector.auth_config 逻辑
- **已有配置数据不受影响**，只是多了一条上下文获取路径

**tenant_id 类型迁移策略（Stage 1 不做类型变更）：**

当前所有表的 `tenant_id` 字段是 `bigint`。河狸云实际企业 ID 是字符串（'RYSGS'）。

**本阶段不改数据库字段类型**，而是：
1. BeaverSessionScope 中 enterprise_id 为字符串
2. 在 scope.py 中建立映射：`enterprise_id('RYSGS') → tenant_id(1)`
3. 引擎内部仍用 int tenant_id 查询配置数据
4. 彻底迁移留到 Stage 3 或之后，届时配置数据量更大、需要正式迁移脚本

**回滚方案：** 删除 kernel/ 目录，chat.py 恢复原签名。无数据迁移需回滚。

### 2.3 Stage 2：Capability 升级（Action → Capability）

**数据库影响：有，新增字段**

| 改造项 | 影响范围 | 兼容性风险 | 保障措施 |
|--------|----------|-----------|----------|
| ai_action 新增 capability_code | 表结构 | 低 | **可空字段，现有数据不受影响** |
| ai_action 新增 input_schema | 表结构 | 低 | 可空 JSON |
| ai_action 新增 output_schema | 表结构 | 低 | 可空 JSON |
| ai_action 新增 policy_config | 表结构 | 低 | 可空 JSON |
| ai_action 新增 side_effect_type | 表结构 | 低 | 默认值 'read' |
| 引擎读取 policy_config | 引擎逻辑 | 中 | 无 policy_config 时跳过策略检查 |

**Alembic 迁移设计原则：**

```sql
-- 所有新字段都是 nullable 或有默认值
ALTER TABLE ai_action ADD COLUMN capability_code VARCHAR(64) NULL;
ALTER TABLE ai_action ADD COLUMN input_schema JSON NULL;
ALTER TABLE ai_action ADD COLUMN output_schema JSON NULL;
ALTER TABLE ai_action ADD COLUMN policy_config JSON NULL;
ALTER TABLE ai_action ADD COLUMN side_effect_type VARCHAR(16) DEFAULT 'read';
```

**兼容性保证：**
- 现有 6 条 Action 记录：新字段为 NULL / 默认值，不影响现有功能
- 引擎逻辑：`if action.policy_config: check_policy(...)` — 空则跳过
- 前端：新字段在管理界面中增加可选编辑区域，不改现有表单
- **capability_code 与 action_code 共存**：旧配置用 action_code，新配置推荐填 capability_code

**回滚方案：** Alembic downgrade + git revert。新增字段为 nullable，downgrade 只是删列，不影响原有数据。

### 2.4 Stage 3：桥接落地 + 客户场景

**数据库影响：可能有，取决于 Capability Gateway 接入方式**

| 改造项 | 影响范围 | 兼容性风险 | 保障措施 |
|--------|----------|-----------|----------|
| Connector 支持 gateway 类型 | ai_connector 表 | 低 | 新增 type 枚举值 |
| 认证模式升级 | connector auth_config | 中 | 保留 proxy_headers，新增 scope_token 模式 |
| tenant_id 类型正式迁移 | 全库 | **高** | 专门的数据迁移脚本 + 完整回归测试 |

**tenant_id bigint → varchar 迁移方案（如果在此阶段执行）：**

```python
# Alembic migration 分三步
def upgrade():
    # Step 1: 新增 varchar 列
    op.add_column('ai_entity', sa.Column('enterprise_code', sa.String(64)))
    # ... 对所有含 tenant_id 的表重复

    # Step 2: 数据迁移（映射表驱动）
    op.execute("""
        UPDATE ai_entity e
        JOIN ai_tenant t ON e.tenant_id = t.id
        SET e.enterprise_code = t.enterprise_code
    """)

    # Step 3: 确认数据完整后，后续 migration 删除旧列（或保留做兼容）
```

**不建议在此阶段直接删除 tenant_id 列**，而是双列共存过渡：
- 新代码读写 enterprise_code
- 旧代码/旧配置仍可通过 tenant_id 工作
- 彻底删除留到 Stage 4 平台化时清理

**回滚方案：** Alembic downgrade 删除新列，数据不丢失。

### 2.5 Stage 4：平台化（Skill 拆分 + Eval + 模板）

**数据库影响：大，但此阶段系统已成熟**

| 改造项 | 影响范围 | 兼容性风险 | 保障措施 |
|--------|----------|-----------|----------|
| Skill → Scenario + Playbook | 表结构重构 | 高 | 数据迁移脚本 + 双表过渡 |
| Eval 体系 | 新增表 | 无 | — |
| 模板引擎 | 新增代码 | 低 | — |

**Skill 拆分迁移策略：**

```
现有 ai_skill 表（9 条记录）
    ↓ 迁移脚本
ai_scenario（场景匹配层）← skill_name, match_keywords, match_patterns, skill_description
ai_playbook（执行编排层）← flow_type, workflow_config, max_tool_calls, response_prompt, response_template
ai_scenario_policy（策略层）← clarification_config, max_response_tokens, summary_threshold, llm_config_id
```

**过渡期设计：**
- 新建 3 张表，从 ai_skill 迁移数据
- ai_skill 表**保留不删**，标记为 deprecated
- 引擎优先读新表，新表无数据时 fallback 读 ai_skill
- 确认所有场景迁移完毕后，未来版本删除 ai_skill

## 3. 跨 Stage 的兼容性红线

### 3.1 绝对不能破坏的

1. **已配置的 9 个 Skill、6 个 Action、6 个 Entity 的运行时行为不能中断**
2. **SSE 流式协议（AG-UI 事件格式）不能变**：前端依赖这个格式
3. **管理后台 API 的入参/出参格式不能变**：前端管理界面依赖
4. **chat API 的调用方式不能变**：`/api/v1/chat/stream` 和 `/api/v1/chat/completions`

### 3.2 可以渐进变更的

1. **引擎内部实现**：只要输入/输出不变，内部怎么重构都行
2. **数据库新增字段**：nullable 或有默认值，不影响现有数据
3. **新增 API 端点**：不影响现有端点
4. **新增模块/目录**：kernel/、tests/ 等

### 3.3 需要迁移脚本的

1. **tenant_id int → enterprise_code varchar**：需要映射表 + 数据迁移
2. **Skill → Scenario + Playbook**：需要拆分脚本 + 双表过渡
3. **Connector auth_config 结构变更**：需要配置迁移脚本

## 4. 每次升级的标准操作流程（SOP）

### 4.1 升级前

```bash
# 1. 备份数据库
mysqldump -h 127.0.0.1 -P 13306 -u root -pbeaver2026 beaver_ai > backup/beaver_ai_$(date +%Y%m%d_%H%M%S).sql

# 2. 记录当前 Alembic 版本
cd backend && source .venv/bin/activate
python -c "from app.database import engine; from sqlalchemy import text; \
  print(engine.connect().execute(text('SELECT version_num FROM alembic_version')).scalar())"

# 3. Git 打 tag
git tag -a stage-X-pre -m "Stage X 升级前快照"
```

### 4.2 升级中

```bash
# 4. 执行 Alembic 迁移
alembic upgrade head

# 5. 运行测试
pytest tests/ -v

# 6. 启动服务验证
uvicorn app.main:app --host 0.0.0.0 --port 8000

# 7. 手动验证关键场景（至少跑通一个完整对话）
```

### 4.3 升级后

```bash
# 8. 记录新 Alembic 版本
# 9. Git commit + tag
git tag -a stage-X-done -m "Stage X 升级完成"
git push origin main --tags
```

### 4.4 回滚

```bash
# 降级数据库
alembic downgrade <previous_version>

# 回退代码
git checkout stage-X-pre

# 或从备份恢复
mysql -h 127.0.0.1 -P 13306 -u root -pbeaver2026 beaver_ai < backup/beaver_ai_YYYYMMDD_HHMMSS.sql
```

## 5. 风险登记簿

| 编号 | 风险 | Stage | 概率 | 影响 | 缓解措施 |
|------|------|-------|------|------|----------|
| R1 | 双引擎合并后行为不一致 | 0 | 中 | 高 | 先补测试建基线，对比双引擎输出 |
| R2 | Scope 注入导致查询逻辑变更 | 1 | 低 | 中 | scope 可选，无 scope 走原逻辑 |
| R3 | 新增字段导致 ORM 加载异常 | 2 | 低 | 中 | 新字段 nullable + 默认值 |
| R4 | tenant_id 迁移数据不一致 | 3 | 中 | 高 | 双列共存过渡，不直接删旧列 |
| R5 | Skill 拆分后旧场景失效 | 4 | 中 | 高 | fallback 读旧表，过渡期双表共存 |
| R6 | 前端管理界面与后端 schema 不匹配 | 2-4 | 低 | 中 | 新字段在前端为可选，不改必填项 |

## 6. 给我们自己的提醒

1. **每个 Stage 开始前必须备份数据库**
2. **每个 Alembic migration 都必须有 downgrade 函数**
3. **新增字段永远是 nullable 或有默认值**，不破坏已有数据
4. **不要在重构过程中顺手改配置数据含义**——结构变、语义不变
5. **引擎对外接口签名变更必须向前兼容**——新增参数必须有默认值
6. **任何迁移脚本先在备份上跑一遍**，确认无误再应用到主库
