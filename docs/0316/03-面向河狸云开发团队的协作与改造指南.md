# 面向河狸云开发团队的协作与改造指南

> 目标：帮助河狸云 Java 团队正确理解 ai-agent 的定位、边界和长期协作方式，避免两边重复建设、职责冲突和后期耦合失控。

## 1. 先统一一个核心认知

### 1.1 ai-agent 不是来替代河狸云主系统的

Java 河狸云仍然是核心主系统，继续负责：
- 主业务流程
- 多租户主数据
- 用户和权限主数据
- 正式业务动作
- 正式存储与审计
- 对外正式业务 API

### 1.2 ai-agent 也不是"外挂聊天页"

ai-agent 的定位应明确为：

**河狸云未来 AI 相关场景的统一智能内核。**

### 1.3 两者关系的正确理解

- Java = System of Record
- Python AI Kernel = System of Intelligence

这两个系统应长期共存，并通过标准协议协作。

---

## 2. 关键事实：AI Kernel 第一阶段不需要 Java 团队改动

这是本次修订的最大变化。

**原来的方案：** 要求 Java 团队提供 `POST /ai/context/exchange` 接口来签发 AI Context Ticket。

**修订后的方案：** 河狸云前端每次请求已经在 Header 中携带了完整的上下文信息（Enterpriseid、Memberid、Orgid、Ouid、Authorization）。AI Kernel 直接从 Header 提取这些字段，组装成 BeaverSessionScope。**Java 团队零改动。**

### 为什么可以这样做

从真实的请求抓包可以看到，河狸云前端每次 API 调用都携带：

| Header | 值 | AI 用途 |
|--------|-----|---------|
| Enterpriseid | RYSGS | 确定当前企业上下文 |
| Memberid | 1583859697503438597 | 确定当前用户在该企业的身份 |
| Orgid | RYSGS | 确定当前组织 |
| Ouid | 495232527520432 | 全局用户唯一 ID |
| Authorization | Bearer eyJ0eXAi… | 代理调用河狸云 API 的凭证 |
| Referer | .../enterprise/RYSGS/approval/audit | 解析当前页面模块 |

这些信息已经足够 AI Kernel 构建完整的会话作用域。

---

## 3. Java 团队需要配合的事项（按阶段）

### 第一阶段（Stage 0-1）：零改动

AI 侧自行完成：
- 从前端代理请求的 Header 提取上下文
- 用 Authorization token 代理调用河狸云 API 获取 rc_enterprise 等完整数据（带缓存）
- 构建 BeaverSessionScope

**Java 团队这个阶段什么都不用做。**

### 第二阶段（Stage 2-3）：Capability Gateway

**时间：** 预计第 9-12 周启动。

#### 需要 Java 团队提供的接口

**1）Capability Gateway**

```
POST /ai/capabilities/execute
```

入参：
```json
{
  "capability_code": "issue.close",
  "scope": {
    "enterprise_id": "RYSGS",
    "member_id": "1583859697503438597",
    "org_id": "RYSGS"
  },
  "params": {
    "issue_id": "12345",
    "close_reason": "已整改完成"
  },
  "request_id": "req_abc123",
  "trace_id": "trace_xyz"
}
```

出参：
```json
{
  "status": "success",
  "result": { ... },
  "evidence": {
    "object_type": "issue",
    "object_id": "12345",
    "status": "closed",
    "timestamps": { "closed_at": "2026-03-16T10:30:00Z" }
  },
  "error_code": null,
  "error_message": null
}
```

**先聚焦 3-5 个核心能力：**
- 问题查询（issue.query）
- 问题关闭（issue.close）
- 巡检查询（inspection.list）
- 验收查询（acceptance.query）
- 概况查询（project.progress.summary）

**2）统一错误码**

| 错误码 | 含义 |
|--------|------|
| no_permission | 无权限 |
| invalid_scope | 作用域无效 |
| invalid_status | 状态不允许此操作 |
| not_found | 对象不存在 |
| need_confirmation | 需要用户确认 |

**3）Metadata 接口**

输出项目/区域/组织/状态字典，供 AI 本体与主系统字典对齐。

### 第三阶段（Stage 4+）：Context Ticket

当 AI 场景越来越多、对安全性要求更高时，升级为 Java 侧签发正式的 AI Context Ticket：

```
POST /ai/context/exchange
```

输入当前登录态，返回短时效的 AI Context Ticket（含 Scope + 允许能力列表 + 过期时间）。但这是远期事项，不是现在。

## 4. 需要 Java 团队确认的技术细节

### 4.1 roleType 位标记含义

rc_enterprise.roleType = 128 是位标记，不是枚举值。AI Kernel 未来做能力权限过滤需要知道各 bit 的含义（如 128 = 0b10000000 可能代表"集团管理员"）。

**请求：** 提供 roleType 各 bit 位的权限说明文档。

### 4.2 企业 ID 的唯一性约束

企业 ID 是字符串编码（如 'RYSGS'、'BLOGI'），与 tenantId 相同。确认：企业编码是否全局唯一？是否会变更？

### 4.3 Rcsign 签名算法

当前 AI Kernel 不验证签名，但第三阶段 Context Ticket 签发时需要。需要了解 Rcsign 的签名算法和密钥管理方式。

## 5. Java 团队长期角色定位

### 角色一：主数据与权限主源维护者
身份、组织、角色、权限、业务数据的权威来源。AI 不应复制这些权威判断。

### 角色二：能力提供者
提供标准能力、标准返回、标准错误码。不是"每来一个 AI 场景就临时补一个接口"。

### 角色三：规则边界维护者
写操作、正式业务状态流转，仍应由主系统裁定。AI 可以理解和发起，但最终规则裁决应在主系统。

---

## 6. Java 团队不应该做的事

1. 不要在 Java 主系统里复制一套 Agent 引擎
2. 不要把大量 prompt 逻辑直接写到 Java 业务代码里
3. 不要让 AI 直接依赖页面 localStorage 结构（AI Kernel 应依赖 Header，不是 localStorage）
4. 不要把所有 AI 需求理解成"加一个聊天入口"
5. 不要让主系统为每个 AI 场景零散补很多临时接口
6. 不要把"实现效果"放在"系统边界"之前

---

## 7. 长期协作模型

### 7.1 推荐分工

| 角色 | 职责 |
|------|------|
| AI Kernel 负责人（你） | 架构、Ontology/Capability/Policy 体系、场景平台化 |
| Java 团队 | 主系统稳定演进、Capability Gateway、权限和状态裁决 |
| 产品/项目团队 | 业务语义梳理、能力配置、场景配置、交付验证 |

### 7.2 协作节奏建议

**双周机制：**
- AI Kernel 能力新增/重构评审
- 河狸云能力网关需求评审
- 当前场景问题复盘
- 下阶段能力清单对齐

### 7.3 变更机制
任何一方修改上下文协议、能力返回结构、权限口径，都必须提前通知并通过测试集验证。

---

## 8. 给 Java 团队的一句结论

你们未来不是在给 AI"让路"，也不是被 AI"替代"。

你们是在和 AI Kernel 分工：你们守住业务事实、权限、流程和正式执行；AI Kernel 负责理解、编排、解释和快速扩展智能场景。

**最正确的长期关系：河狸云主系统稳，AI Kernel 快；主系统可信，AI 系统可扩；两者解耦协作，不互相吞没。**

而且，**第一阶段 Java 团队零改动**——AI 侧自行从现有 Header 中获取所需的一切上下文。
